Q2c
(i) Risk Identification & (ii) PERT Technique

(i) Risk Identification

Risk identification is the process of determining which risks may affect the project and documenting their characteristics.

Methods:

  • Brainstorming — team discussions to list potential risks
  • Checklists — past project risk lists
  • Delphi Technique — expert consensus anonymously
  • SWOT Analysis — Strengths, Weaknesses, Opportunities, Threats
  • Assumption Analysis — test validity of project assumptions

Risk Categories:

CategoryExamples
TechnicalNew technology, integration failures, complexity
ProjectSchedule slippage, resource unavailability
BusinessMarket change, budget cuts, stakeholder changes
ExternalVendor failure, regulatory changes

Output: Risk Register — lists risks with likelihood, impact, and owners.

(ii) PERT Technique

PERT (Program Evaluation and Review Technique) is a probabilistic scheduling method for projects where activity durations are uncertain.

Expected Time: te = (O + 4M + P) / 6
Variance: σ² = [(P − O) / 6]²
Where O=Optimistic, M=Most Likely, P=Pessimistic

Q5a
Project Scheduling Techniques & Network Models

Scheduling Techniques:

TechniqueTypeBest For
Gantt ChartBar chartSimple visualization; task durations
CPMNetwork (deterministic)Fixed durations; find critical path
PERTNetwork (probabilistic)Uncertain durations; R&D projects
WBSHierarchyBreaking scope into manageable tasks
Resource HistogramBar chartResource utilization over time
Milestone ChartSimple timelineExecutive reporting

How Network Models Are Used:

  1. List all activities with durations and dependencies
  2. Draw network diagram (AON — Activity on Node)
  3. Forward Pass → ES (Earliest Start) and EF (Earliest Finish)
  4. Backward Pass → LS (Latest Start) and LF (Latest Finish)
  5. Float = LS − ES (zero float = critical activity)
  6. Critical Path = longest path = minimum project duration

Critical Path Method (CPM)

Key Concepts:

TermFormulaMeaning
ES (Earliest Start)From forward passEarliest an activity can begin
EF (Earliest Finish)ES + DurationEarliest an activity can end
LS (Latest Start)From backward passLatest start without delaying project
LF (Latest Finish)LS + DurationLatest end without delaying project
Total FloatLS − ES = LF − EFSlack available; 0 = critical

Activity Node Structure:

┌──────────────────────────┐ │ ES │ Duration │ EF │ │──────────────────────────│ │ Activity Name │ │──────────────────────────│ │ LS │ Float │ LF │ └──────────────────────────┘

How CPM Identifies Bottlenecks:

  • Activities on critical path are bottlenecks — delay = project delay
  • Near-critical activities (low float) need close monitoring
  • Resource leveling — shift resources from high-float to critical activities
  • Crashing — add resources to critical path to shorten duration
  • Fast-tracking — do critical activities in parallel

Define Project with Example

A project is a temporary, unique endeavor undertaken to create a specific product, service, or result with a defined beginning and end.

Limitations of Waterfall Model
  1. No flexibility — requirements must be fixed upfront; poor for evolving needs
  2. Late testing — defects found late are exponentially more expensive to fix

Q1d
Parametric Model for Effort Estimation

A parametric model uses mathematical relationships between project parameters and effort.

General form: Effort = a × (Size)^b × Adjustment Factors

Examples:

  • COCOMO Basic: Effort = 2.4 × KLOC^1.05 (Organic mode)
  • COCOMO II: PM = 2.94 × Size^E × ∏EM
  • Function Point-based: Effort = FP × productivity_factor

Parameters used: lines of code, function points, complexity, team capability.

Q2a
Why DCF is Better than Net Profit or ROI for Project Selection

Core Issue — Time Value of Money:

₹1 today is worth more than ₹1 in 5 years because money can be invested. Net Profit and ROI ignore when money is received.

Limitations of Net Profit:

  • Simply sums inflows − outflows
  • Treats ₹10L in year 1 the same as ₹10L in year 10
  • Cannot compare projects with different durations

Limitations of ROI:

  • ROI = (Total Profit / Investment) × 100
  • 50% ROI over 10 years may be worse than 40% ROI over 3 years

Why DCF / NPV is Superior:

PV = CF_t / (1 + r)^t NPV = Σ PV − Initial Investment
  • Discounts future cash flows to present value
  • Projects with positive NPV create value; higher NPV = better project
  • Accounts for: inflation, opportunity cost, risk (via discount rate)
  • Allows comparison of projects with different time horizons
Rule: Accept projects with NPV > 0. Between alternatives, choose highest NPV.

Q3a
Project Ahead of Schedule — Which Option to Choose?

Situation: 3 months ahead, 15% under budget. Options:

  1. Finish early; release team; return budget
  2. Use time & budget for more testing
  3. Use time & budget for better documentation

Analysis:

OptionProsCons
1 — Finish earlyDemonstrates efficiency; releases teamMissed quality improvement opportunity
2 — More testing ✓Catches residual defects; reduces post-release cost 10–100×Diminishing returns if already well-tested
3 — Better documentationReduces long-term maintenance costLess immediate quality impact
Recommended: Option 2 (More Testing)
Boehm's principle: defect removal before release is cheapest. Every complex system has latent defects. Pareto principle: find the remaining 20% of bugs causing 80% of failures. Option 2 maximizes long-term value.
A hybrid of Options 2+3 is also defensible.

ETC = (BAC − EV) / (CPI × SPI) = (22,000 − 13,000) / (0.867 × 0.929) = 9,000 / 0.806 = $11,166

Step 3: EAC (Estimate at Completion):

EAC = AC + ETC = 15,000 + 11,166 = $26,166

Step 4: VAC (Variance at Completion):

VAC = BAC − EAC = 22,000 − 26,166 = −$4,166
Interpretation: Project will overrun by $4,166. Both CPI and SPI < 1 indicate the project is over budget AND behind schedule. Immediate corrective action required.

Q2b
COSMIC Full Function Points for Software Cost Estimation

COSMIC (Common Software Measurement International Consortium) measures software functional size through counting data movements.

Four Data Movement Types (each = 1 CFP):

TypeSymbolDirectionSPM Example
EntryEData moves INTO software from userStudent submits registration form
ExitXData moves OUT of software to userSystem displays grade report
ReadRSoftware reads from persistent storageFetching course list from DB
WriteWSoftware writes to persistent storageSaving exam result to DB

Measurement Process:

  1. Identify functional users (humans or systems interacting with software)
  2. Identify functional processes (requirements that trigger processing)
  3. For each process, count all data movements (E, X, R, W)
  4. Total CFPs = total data movements

Worked Example — Student Registration:

ActionTypeCFPs
Student enters registration dataE1
System reads course availability from DBR1
System saves registration to DBW1
System displays confirmation to studentX1
Total for this process4 CFP
Effort = Total CFP × (hours per CFP from historical productivity data)

Advantages over Function Point Analysis:

  • Handles real-time and embedded systems better
  • More objective and repeatable
  • Language and technology independent
  • Simpler counting rules
